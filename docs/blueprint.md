# **App Name**: Simpliwork Deal OS v1

## Core Features:

- Role-Based Access Control: Secure user login and access control, routing users to specific interfaces based on their assigned role (Sales, Design, Management, Admin). Ensures authenticated access and enforces permissions for viewing and modifying data based on user role, including restricted visibility for Design (only Solutioning deals) and read-only access for Management.
- Sales Deal Management: Empowers Sales users with a dedicated 'My Deals' homepage. Enables creating new deals by capturing mandatory fields (company_name, building_id, sales_owner_email, requirement_summary, source_type, source_name). Allows Sales to view, open, and update deal details, including qualification fields, sales/commercial signals (budget_clarity, timeline_clarity, decision_maker_identified), and critical dates (last_activity_date, loi_initiated_date, loi_signed_date). Supports manual stage progression via dropdown for 'Negotiation', 'LoI Initiated', 'LoI Signed', and 'Lost', with mandatory reasons for 'Lost' and required loi_signed_date for 'LoI Signed'.
- Design Solutioning Workflow: Provides Design users with a 'Solutioning Queue' homepage, displaying only deals in the 'Solutioning' stage. Design users can update specific deal fields: layout_uploaded_date, increment/edit layout_revision_count, add design_note, and upload layout_file_upload. Design users are restricted from changing deal stages or editing sales/commercial/LoI/lost fields.
- Management Pipeline Overview (God View): Offers a read-only 'God View' dashboard for Management, presenting all deals across the pipeline. Includes comprehensive filtering by building, cluster, stage, sales owner, source type, source name, and creation date range. Features key performance indicators such as 'active deals by building', 'deals by stage', 'LoI Initiated count', 'LoI Signed count', 'Lost count', 'stuck deals', 'no recent activity', and 'high revision-count deals'. Supports cohort analysis for deals created within specific timeframes and their current status.
- Master Data & User Administration: Dedicated admin pages for managing core master data: Buildings (building_id, building_name, city, cluster, active_status) and Users (user_id, full_name, email, role, active_status). Admin can perform backend-authenticated corrections and manage active statuses for buildings and users. This role has no rich daily workflow beyond master data management.
- Deal Stage Enforcement & Validation: Automated backend logic to validate and enforce all defined deal stage progression rules. Stages include: Qualified, Solutioning, Proposal Sent, Negotiation, LoI Initiated, LoI Signed, Lost. Validation rules: Creating a deal requires company_name, building_id, sales_owner_email, requirement_summary, source_type, source_name. Moving to 'Proposal Sent' requires layout_uploaded_date OR layout_file_upload. Moving to 'LoI Signed' requires loi_signed_date. Moving to 'Lost' requires lost_reason. Prevents unauthorized or incomplete stage transitions and ensures data integrity.
- Robust Data Architecture: Secure and modular data persistence layer designed to store and manage all deal, building, and user data according to strict schemas. Ensures data integrity and scalability, allowing for future module additions (e.g., site visits, legal workflow, document vault) without architectural overhaul. All fields including deal_id, company_name, building_id, sales_owner_email, stage, requirement_summary, approx_requirement_size, source_type, source_name, created_date, last_activity_date, budget_clarity, timeline_clarity, decision_maker_identified, layout_requested_date, layout_uploaded_date, layout_revision_count, design_note, layout_file_upload, loi_initiated_date, loi_signed_date, and lost_reason for Deals; building_id, building_name, city, cluster, active_status for Buildings; and user_id, full_name, email, role, active_status for Users are strictly adhered to.

## Style Guidelines:

- A clean, professional light color scheme to convey efficiency and focus. The primary brand color is a deep, trustworthy blue (#2259C3). This will be used for interactive elements and key headers, standing out effectively against the light background. The background color is a subtle, almost-white blue-grey (#F0F2F5), ensuring readability and reducing eye strain during long usage. An accent color, a soft and light cyan-blue (#85D3E0), will highlight critical information or provide visual cues for action.
- The entire application will use 'Inter', a modern grotesque-style sans-serif font. Its neutral and objective design provides excellent legibility for both headlines and detailed data tables, aligning perfectly with the app's professional and minimalistic aesthetic for internal enterprise use.
- Use a set of minimal, outline-style icons to maintain visual clarity and reduce clutter. Icons should be clear and immediately understandable, complementing the clean UI without drawing undue attention.
- Prioritize a grid-based, adaptive layout for minimal clutter and fast navigation, especially for role-based homepages and detail views. Dashboards will feature a card-based structure for clear data presentation and analytical views. Forms, such as 'Create Deal' and 'Deal Detail', will be streamlined with logical groupings to support 'Sales-first ergonomics' and efficient data entry and updates. Role-based navigation will be clear and intuitive.
- Implement subtle and purposeful micro-animations for transitions and user feedback. These will include smooth loading states, button hover effects, and minimal transitions between pages, enhancing the perception of speed and responsiveness without distracting from the data.
